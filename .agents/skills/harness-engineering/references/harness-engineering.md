# Harness Engineering Reference

Source: OpenAI, "Harness engineering: leveraging Codex in an agent-first world", Ryan Lopopolo, February 11, 2026.
URL: https://openai.com/index/harness-engineering/

This reference is a structured condensation of the article for use by Codex. It is not a verbatim copy. Use it when a task needs the article's broader context, examples, or rationale beyond the short workflow in `SKILL.md`.

## Thesis

Harness engineering is the practice of designing the environment around coding agents so they can perform reliable software engineering work with limited human attention. The core shift is from humans primarily writing code to humans designing systems of intent, context, tools, validation, review, and feedback loops.

The article's operating summary: humans steer, agents execute.

## Experiment Context

OpenAI describes an internal experiment where a small team built and shipped an internal beta product with no manually written code. Codex wrote application logic, tests, CI configuration, documentation, observability, internal tooling, and repository-management scripts.

Key reported details:

- The first commit landed in late August 2025.
- The initial repository scaffold was generated by Codex CLI using GPT-5 and existing templates.
- The initial `AGENTS.md` was also generated by Codex.
- After roughly five months, the repo contained around one million lines across code, infrastructure, tooling, docs, and utilities.
- Around 1,500 pull requests were opened and merged.
- The team started with three engineers driving Codex and later grew to seven.
- The product had internal daily users and external alpha testers.
- The team estimates the work took about one tenth of the time manual implementation would have taken.

These numbers are contextual evidence, not a recipe. The reusable lesson is that code throughput was enabled by a harness: repo structure, tools, feedback loops, documentation, constraints, and review automation.

## Redefining Engineering Work

The article frames the engineer's role as moving up a level of abstraction.

In a no-manual-code workflow, the engineer does not patch around agent limitations by writing the missing code. Instead, the engineer asks what capability is missing from the harness:

- Is the task underspecified?
- Is important context hidden outside the repo?
- Is the app not observable enough for the agent?
- Is a validation loop missing?
- Is an architectural rule only documented in prose?
- Is a human repeatedly giving the same review feedback?

The answer should usually become a repo-local artifact: a doc, script, test, lint rule, skill, validation tool, or review loop.

### Depth-First Development

The team worked depth-first: break large goals into smaller building blocks, have Codex build those blocks, then use the new capabilities to unlock harder tasks.

Reusable pattern:

1. Identify the high-level product or engineering goal.
2. Break it into design, code, review, test, observability, and repair pieces.
3. Ask Codex to build each piece.
4. Use the new piece as infrastructure for the next piece.
5. Convert failures into harness improvements.

### Agent Review Loop

The article describes a PR-completion loop where Codex:

- Reviews its own changes locally.
- Requests specific agent reviews locally and in the cloud.
- Responds to human and agent feedback.
- Iterates until reviewers and checks are satisfied.
- Uses standard tools such as `gh`, local scripts, and repo-embedded skills to gather context.

The direction of travel is toward agent-to-agent review, with humans increasingly reserved for judgment and prioritization.

## Increasing Application Legibility

As code throughput rose, human QA became a bottleneck. The team responded by making the application itself legible to Codex.

Agent legibility means the agent can directly inspect and reason about runtime behavior without a human manually copying observations into the prompt.

### UI Legibility

The app was made bootable per git worktree so each task could launch and drive an isolated app instance. Chrome DevTools Protocol was connected to the agent runtime. Skills were created for:

- DOM snapshots
- Screenshots
- Navigation
- UI state before and after interactions
- Runtime event observation

This let Codex reproduce bugs, validate fixes, and reason about UI behavior directly.

### Observability Legibility

Logs, metrics, and traces were exposed through a local, ephemeral observability stack per worktree. Codex could query these signals through APIs and use them during debugging and validation.

Examples of tractable prompts once observability is agent-readable:

- Ensure service startup completes under a target latency.
- Ensure critical user journeys have no slow spans above a threshold.
- Correlate logs, metrics, traces, and UI behavior after a code change.

### Long-Running Agent Work

The article notes that single Codex runs sometimes worked for hours on one task. Long-running autonomy only works when the harness supplies enough visibility and feedback for the agent to keep making progress without human observation.

## Repository Knowledge As System Of Record

Context management is a central problem for large agent tasks. The article's lesson: give Codex a map, not a giant instruction manual.

### Why A Giant AGENTS.md Fails

A monolithic instruction file fails because:

- Context is scarce, and a giant file crowds out task-specific context.
- Excessive guidance becomes indistinct; everything cannot be equally important.
- Large manuals rot quickly as rules become stale.
- A single blob is hard to verify for ownership, freshness, cross-links, and coverage.

### Better Pattern

Use a short `AGENTS.md` as a table of contents. Keep deeper knowledge in structured repo-local docs.

Representative layout from the article:

```text
AGENTS.md
ARCHITECTURE.md
docs/
  design-docs/
    index.md
    core-beliefs.md
  exec-plans/
    active/
    completed/
    tech-debt-tracker.md
  generated/
    db-schema.md
  product-specs/
    index.md
    new-user-onboarding.md
  references/
    design-system-reference-llms.txt
    nixpacks-llms.txt
    uv-llms.txt
  DESIGN.md
  FRONTEND.md
  PLANS.md
  PRODUCT_SENSE.md
  QUALITY_SCORE.md
  RELIABILITY.md
  SECURITY.md
```

The important pattern is not the exact filenames. The important pattern is progressive disclosure:

- `AGENTS.md` is the stable entry point.
- Index files tell the agent where to go next.
- Design docs capture architectural decisions and core beliefs.
- Execution plans capture complex work, progress, and decisions.
- Generated docs expose schemas or external references in agent-readable form.
- Quality, reliability, security, frontend, design, and product docs define evaluation standards.

### Mechanical Freshness

The knowledge base should be enforced mechanically. The article mentions linters and CI jobs that validate structure, cross-links, and freshness. It also mentions recurring doc-gardening agents that scan for stale docs and open fix-up PRs.

## Agent Legibility As A Design Goal

The repository should be optimized for Codex's ability to reason about the business and technical domain.

Anything the agent cannot access during its run effectively does not exist. This includes knowledge in chat threads, external docs, and human memory. If a Slack discussion decides an architectural pattern, that decision should be moved into the repository.

### Technology Choices

The article argues for dependencies and abstractions that the agent can inspect, validate, and modify. Boring technologies can be beneficial because they are stable, composable, and well represented in training data.

Sometimes it can be cheaper to implement a small, inspectable internal helper than to depend on opaque upstream behavior. The article gives the example of implementing a map-with-concurrency helper rather than pulling in a generic package, because the internal helper could be integrated with OpenTelemetry, fully tested, and tailored to runtime expectations.

This is not an argument against dependencies. It is an argument for choosing dependencies and abstractions with agent legibility in mind.

## Enforcing Architecture And Taste

Documentation alone does not keep an agent-generated codebase coherent. The article emphasizes enforcing invariants instead of micromanaging implementation details.

### Architecture Constraints

The described repo uses a rigid layered architecture for each business domain. Dependencies are allowed only in specific directions. Cross-cutting concerns enter through explicit provider interfaces.

Example layer direction from the article:

```text
Types -> Config -> Repo -> Service -> Runtime -> UI
```

Cross-cutting concerns such as auth, connectors, telemetry, and feature flags should enter through explicit interfaces rather than arbitrary imports.

### Mechanical Enforcement

Rules are enforced with custom linters and structural tests. Examples of enforceable rules include:

- Dependency direction between layers
- Allowed cross-domain edges
- Structured logging requirements
- Schema and type naming conventions
- File size limits
- Platform-specific reliability requirements
- Boundary validation for external data shapes

The article highlights writing custom lint messages so they provide remediation instructions to agents. This turns an error into useful context.

### Taste As System Feedback

Human taste is fed back into the system through review comments, refactoring PRs, user-facing bugs, docs, and tooling. When a rule recurs often enough, promote it from prose to a mechanical check.

The goal is to enforce boundaries centrally while allowing local autonomy inside those boundaries.

## Throughput Changes Merge Philosophy

High agent throughput changes the cost tradeoffs around blocking gates.

The article describes an environment with:

- Minimal blocking merge gates
- Short-lived PRs
- Follow-up runs for some flakes rather than indefinite blocking
- Cheap corrections compared with expensive waiting

This is explicitly contextual. The article notes that this would be irresponsible in lower-throughput environments. The reusable principle is to tune merge strategy to the cost of waiting, the cost of correction, and the maturity of the validation harness.

## What Agent-Generated Means

In the article, agent-generated means the agent creates all repository artifacts, not only product code.

Agents produce:

- Product code
- Tests
- CI configuration
- Release tooling
- Internal developer tools
- Documentation
- Design history
- Evaluation harnesses
- Review comments and responses
- Repository-management scripts
- Production dashboard definitions

Humans remain in the loop, but they operate at the layer of prioritization, feedback translation, acceptance criteria, outcome validation, and judgment.

## Increasing Autonomy

The article describes autonomy increasing as more of the loop is encoded into the system: testing, validation, review, feedback handling, and failure recovery.

A mature harness can let Codex perform an end-to-end feature or bug-fix loop from a single prompt:

1. Validate current repo state.
2. Reproduce the reported bug.
3. Record evidence of the failure, such as a video.
4. Implement a fix.
5. Validate by driving the app.
6. Record evidence of the resolution.
7. Open a PR.
8. Respond to agent and human feedback.
9. Detect and remediate build failures.
10. Escalate only when judgment is required.
11. Merge the change.

The article cautions that this depends heavily on the specific repo structure and tooling. Do not assume this level of autonomy generalizes without similar investment.

## Entropy And Garbage Collection

Agent autonomy introduces drift. Codex tends to reproduce existing patterns, including uneven or suboptimal ones. Without cleanup, bad patterns compound.

The team initially handled cleanup manually and found it did not scale. The improved approach was to encode golden principles into the repository and run recurring cleanup processes.

Examples of golden principles from the article:

- Prefer shared utility packages over repeated hand-rolled helpers, so invariants stay centralized.
- Do not probe data shapes by guessing; validate boundaries or rely on typed SDKs.

Recurring background Codex tasks scan for deviations, update quality grades, and open targeted refactoring PRs. Many can be reviewed quickly and automerged.

The article compares this to garbage collection: technical debt compounds, so pay it down continuously in small increments.

## Open Questions

The article says the approach worked through internal launch and adoption, but several long-term questions remain:

- How does architectural coherence evolve over years in a fully agent-generated system?
- Where does human judgment add the most leverage?
- How should human judgment be encoded so it compounds?
- How will the harness evolve as models become more capable?

The closing lesson is that software still requires discipline, but more of that discipline moves into scaffolding, tools, abstractions, control systems, and feedback loops.

## Practical Checklist

Use this checklist when applying the article to a repository:

- Is `AGENTS.md` short and navigational?
- Are product rules, architecture, design decisions, plans, schemas, quality bars, and tech debt tracked in repo-local files?
- Can an agent boot the app in an isolated worktree?
- Can an agent inspect UI state through browser automation, DOM snapshots, screenshots, or recordings?
- Can an agent query logs, metrics, and traces locally?
- Are important boundaries enforced by tests, lints, or CI rather than prose only?
- Do lint/test failures include remediation guidance?
- Can agents self-review and request targeted agent reviews?
- Can agents reproduce bugs and capture before/after evidence?
- Can agents recover from build failures and flaky tests?
- Are recurring human review comments converted into docs, checks, or cleanup tasks?
- Is there a recurring doc-gardening or quality-gardening process?
- Are merge gates calibrated to throughput, risk, and correction cost?
- Are humans reserved for prioritization, acceptance criteria, outcome validation, and judgment?
